Decorations are view-only markers attached to positions or ranges in the document. They are declared by extensions and never modify or serialize the document. For concepts, examples, and guidance on when to use decorations, read the [Decorations guide](/editor/core-concepts/decorations). This page is the API reference.

## `addDecorations`

A lifecycle hook on every extension (including nodes and marks). It returns a `DecorationSpec` describing the decorations the extension contributes, or `null` for none. All declarations from all extensions are aggregated into a single ProseMirror plugin by Tiptap's internal decoration manager.

`addDecorations()` is bound to `this` (`{ name, options, storage, editor, type, parent }`) like other extension hooks.

```ts
import { decoration, Extension } from '@tiptap/core'

const MyExtension = Extension.create({
  name: 'myExtension',

  addDecorations() {
    return {
      create: ({ editor, state, view }) => [decoration.inline(1, 5, { class: 'highlight' })],
    }
  },
})
```

### `DecorationSpec`

`DecorationSpec` is a discriminated union selected by its `update` strategy. Every strategy uses `create` for initialization and forced updates.

| Property        | Type                                                                   | Description                                                                                                         |
| --------------- | ---------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------- |
| `update`        | `'document' \| 'changedRanges' \| 'manual'`                            | Controls how decorations update. Defaults to `'document'`.                                                          |
| `create`        | `(props: { editor; state; view }) => DecorationDescriptor[]`           | **Required.** Builds the full set. Used for initialization and `updateDecorations()`.                               |
| `shouldUpdate`  | `(props: { editor; tr; oldState; newState }) => boolean`               | Optional for `document` and `changedRanges`. Return `false` to map without rebuilding. Defaults to `tr.docChanged`. |
| `createInRange` | `(props: { editor; state; view; from; to }) => DecorationDescriptor[]` | Required only for `changedRanges`. Returns decorations anchored within the supplied block-aligned range.            |

```ts
interface BaseDecorationSpec {
  // Build the FULL set of this extension's decorations from the current state.
  // Used for the initial build and for forced updateDecorations().
  create: (props: {
    editor: Editor
    state: EditorState
    view: EditorView
  }) => DecorationDescriptor[]

  // Gate recomputation for performance. Return false to keep existing decorations
  // moved to their new positions instead of rebuilt.
  // Defaults to recomputing whenever the document changed (tr.docChanged).
  shouldUpdate?: (props: {
    editor: Editor
    tr: Transaction
    oldState: EditorState
    newState: EditorState
  }) => boolean
}

// Default: every doc change rebuilds the whole set with create(). Always correct.
interface DocumentDecorationSpec extends BaseDecorationSpec {
  update?: 'document'
  createInRange?: never
}

interface ChangedRangesDecorationSpec extends BaseDecorationSpec {
  update: 'changedRanges'
  // Must return ONLY decorations whose anchor lies within [from, to)
  // (`from` for inline/node decorations, `pos` for widgets).
  createInRange: (props: {
    editor: Editor
    state: EditorState
    view: EditorView
    from: number
    to: number
  }) => DecorationDescriptor[]
}

interface ManualDecorationSpec extends BaseDecorationSpec {
  update: 'manual'
  shouldUpdate?: never
  createInRange?: never
}

type DecorationSpec = DocumentDecorationSpec | ChangedRangesDecorationSpec | ManualDecorationSpec
```

### `DecorationUpdateStrategy`

```ts
type DecorationUpdateStrategy = 'document' | 'changedRanges' | 'manual'
```

- `document` rebuilds the full set with `create` after document changes. This is the default and is always correct.
- `changedRanges` maps existing decorations and calls `createInRange` only for changed, block-aligned ranges. Use it only for block-local decorations.
- `manual` keeps the current decorations until you call `updateDecorations()`.

You can use `shouldUpdate` with `document` and `changedRanges`. You cannot use it with `manual`, because manual decorations only update when you ask them to.

<Callout title="createInRange contract" variant="warning">
  `createInRange` must return **only** decorations whose start position lies within `[from, to)`.
  Use `from` for inline and node decorations, and `pos` for widget decorations. Decorations outside
  this range are ignored. If a decoration depends on content outside the supplied range, use the
  `document` strategy instead.
</Callout>

See [Performance](/editor/core-concepts/decorations#performance) for strategy selection and changed-range correctness constraints.

## `decoration`

Import `decoration` from `@tiptap/core`. Its methods return plain objects that describe what Tiptap should show. Return these objects from `create` and `createInRange`.

All three return a `DecorationDescriptor` (a member of the `DecorationDescriptor` union).

| Method              | Returns                      | Description                                                            |
| ------------------- | ---------------------------- | ---------------------------------------------------------------------- |
| `decoration.node`   | `NodeDecorationDescriptor`   | Adds attributes to a single node's DOM wrapper.                        |
| `decoration.inline` | `InlineDecorationDescriptor` | Wraps a range of inline content in a styled span.                      |
| `decoration.widget` | `WidgetDecorationDescriptor` | Inserts a widget (a DOM node) at a single position. `key` is required. |

### Descriptor types

```ts
interface NodeDecorationDescriptor {
  kind: 'node'
  from: number
  to: number
  attrs: DecorationAttrs
  spec?: Record<string, any>
}

interface InlineDecorationDescriptor {
  kind: 'inline'
  from: number
  to: number
  attrs: DecorationAttrs
  spec?: Record<string, any>
}

interface WidgetDecorationDescriptor {
  kind: 'widget'
  pos: number
  render: (view: EditorView, getPos: () => number | undefined) => HTMLElement
  key: string
  spec?: WidgetDecorationOptions & Record<string, any>
}

type DecorationDescriptor =
  | NodeDecorationDescriptor
  | InlineDecorationDescriptor
  | WidgetDecorationDescriptor
```

Tiptap converts these objects to ProseMirror decorations for you.

### `decoration.node`

```ts
decoration.node(
  from: number,
  to: number,
  attrs?: DecorationAttrs,
  spec?: Record<string, any>,
)
```

**Arguments**

- `from` â€“ start position of the node.
- `to` â€“ end position of the node.
- `attrs` â€“ optional. `DecorationAttrs`: `class`, `style`, `nodeName`, and any other HTML attribute.
- `spec` â€“ optional ProseMirror decoration spec.

```ts
decoration.node(pos, pos + node.nodeSize, { class: 'is-heading' })
```

### `decoration.inline`

```ts
decoration.inline(
  from: number,
  to: number,
  attrs?: DecorationAttrs,
  spec?: Record<string, any>,
)
```

**Arguments**

- `from` â€“ start of the inline range.
- `to` â€“ end of the inline range.
- `attrs` â€“ optional. `DecorationAttrs`: `class`, `style`, `nodeName`, and any other HTML attribute.
- `spec` â€“ optional ProseMirror decoration spec.

```ts
decoration.inline(matchFrom, matchTo, { class: 'is-match' })
```

### `decoration.widget`

```ts
decoration.widget(
  pos: number,
  render: (view: EditorView, getPos: () => number | undefined) => HTMLElement,
  options: {
    key: string
    side?: number
    relaxedSide?: boolean
    marks?: readonly Mark[]
    stopEvent?: (e: Event) => boolean
    ignoreSelection?: boolean
    destroy?: (node: Node) => void
    [k: string]: any
  },
)
```

**Arguments**

- `pos` â€“ the position to render the widget at.
- `render` â€“ a callback that receives the `view` and a `getPos` accessor and returns an `HTMLElement`.
- `options.key` â€“ **required.** A stable, position-independent, globally unique key. See [Widget keys](/editor/core-concepts/decorations#widget-keys). Tiptap logs a development warning when it detects two widget decorations with the same key in a single build.
- `options.side` â€“ optional. Which side of the position the widget biases to (see ProseMirror's [`Decoration.widget`](https://prosemirror.net/docs/ref/#view.Decoration%5Ewidget)).
- `options.relaxedSide` â€“ optional. Allow the DOM selection to remain on either side of the widget.
- `options.marks` â€“ optional. Marks applied to the widget.
- `options.stopEvent` â€“ optional. Return `true` to stop ProseMirror from handling an event originating in the widget.
- `options.ignoreSelection` â€“ optional. Prevent selection changes inside the widget from being synchronized by ProseMirror.
- `options.destroy` â€“ optional. Called with the widget DOM node when ProseMirror removes it or destroys the editor.

```ts
decoration.widget(
  matchFrom,
  () => {
    const el = document.createElement('span')
    el.textContent = 'â˜…'
    return el
  },
  // A position-based key is fine here because this widget has no state.
  { key: `marker-${matchFrom}`, side: -1 },
)
```

<Callout title="Widget keys must be stable and unique" variant="warning">
  ProseMirror reuses a widget's DOM across redraws only when the key matches the previous render,
  and keys must be globally unique. Duplicate or position-derived keys cause flicker, lost state, or
  crashes. See [Widget keys](/editor/core-concepts/decorations#widget-keys) for the full rules.
</Callout>

## `updateDecorations`

This command builds the decorations again for the full document. It ignores the extension's update strategy and `shouldUpdate` for this update. Use it after changing a value outside the document, such as a search term.

```ts
editor.commands.updateDecorations() // Update every extension.
editor.commands.updateDecorations('myExtension') // Update one extension.
```

**Arguments**

- `extensionName` is optional. Pass an extension name to update only that extension. Leave it out to update every extension.

See the [runtime parameters via storage](/editor/core-concepts/decorations#runtime-parameters-via-storage) pattern for the common use case.

## `ReactWidgetRenderer`

Import: `import { ReactWidgetRenderer } from '@tiptap/react'`. Returns a `WidgetDecorationDescriptor` you return from `create` or `createInRange`. It reuses Tiptap's `ReactRenderer` portal infrastructure, so the component lives in the editor's React tree (context and hooks work as usual). Your component additionally receives `editor` and `getPos` as props.

```ts
ReactWidgetRenderer<P>(
  component: ComponentType<P & { editor: Editor; getPos: () => number | undefined }>,
  options: {
    editor: Editor
    pos: number
    key: string
    props?: P
    as?: string
    className?: string
    side?: number
    relaxedSide?: boolean
    marks?: readonly Mark[]
    stopEvent?: (event: Event) => boolean
    ignoreSelection?: boolean
    destroy?: (node: Node) => void
  },
): WidgetDecorationDescriptor
```

| Option            | Type                        | Description                                                                           |
| ----------------- | --------------------------- | ------------------------------------------------------------------------------------- |
| `editor`          | `Editor`                    | The editor instance.                                                                  |
| `pos`             | `number`                    | The document position to render at.                                                   |
| `key`             | `string`                    | Stable, unique key. See [Widget keys](/editor/core-concepts/decorations#widget-keys). |
| `props`           | `P`                         | Props for your component.                                                             |
| `as`              | `string`                    | Wrapper tag, defaults to `'span'`.                                                    |
| `className`       | `string`                    | Class applied to the wrapper element.                                                 |
| `side`            | `number`                    | ProseMirror widget side bias.                                                         |
| `relaxedSide`     | `boolean`                   | Allow the DOM selection on either side of the widget.                                 |
| `marks`           | `readonly Mark[]`           | Marks applied to the widget.                                                          |
| `stopEvent`       | `(event: Event) => boolean` | Stop selected widget events from reaching ProseMirror.                                |
| `ignoreSelection` | `boolean`                   | Prevent selection changes inside the widget from syncing.                             |
| `destroy`         | `(node: Node) => void`      | Run custom cleanup after the framework renderer is removed.                           |

<Callout title="Props and renderer caching" variant="info">
  Fresh `props` are pushed each time `create`/`createInRange` re-runs (ProseMirror skips `toDOM`
  when it reuses the DOM, so props can't ride along there). The renderer for a key is cached and
  reused so component state survives across edits. The cache is swept when the editor is destroyed.
</Callout>

Follow the [React decorations tutorial](/guides/decorations-react) for a complete stateful widget.

## `VueWidgetRenderer`

Import: `import { VueWidgetRenderer } from '@tiptap/vue-3'` (or `'@tiptap/vue-2'`). Same idea as the React renderer; it reuses Tiptap's `VueRenderer`, so the component shares the editor's app context (`provide`/`inject` work). The component receives `editor` and `getPos` in addition to your `props`.

```ts
VueWidgetRenderer(
  component: Component,
  options: {
    editor: Editor
    pos: number
    key: string
    props?: Record<string, any>
    side?: number
    relaxedSide?: boolean
    marks?: readonly Mark[]
    stopEvent?: (event: Event) => boolean
    ignoreSelection?: boolean
    destroy?: (node: Node) => void
  },
): WidgetDecorationDescriptor
```

| Option            | Type                        | Description                                                                           |
| ----------------- | --------------------------- | ------------------------------------------------------------------------------------- |
| `editor`          | `Editor`                    | The editor instance.                                                                  |
| `pos`             | `number`                    | The document position to render at.                                                   |
| `key`             | `string`                    | Stable, unique key. See [Widget keys](/editor/core-concepts/decorations#widget-keys). |
| `props`           | `Record<string, any>`       | Props for your component.                                                             |
| `side`            | `number`                    | ProseMirror widget side bias.                                                         |
| `relaxedSide`     | `boolean`                   | Allow the DOM selection on either side of the widget.                                 |
| `marks`           | `readonly Mark[]`           | Marks applied to the widget.                                                          |
| `stopEvent`       | `(event: Event) => boolean` | Stop selected widget events from reaching ProseMirror.                                |
| `ignoreSelection` | `boolean`                   | Prevent selection changes inside the widget from syncing.                             |
| `destroy`         | `(node: Node) => void`      | Run custom cleanup after the framework renderer is removed.                           |

<Callout title="Vue-specific requirements" variant="warning">
  The component **must render a single root element**. The `editor` is passed raw (with `markRaw`)
  on purpose. Do not wrap it in reactivity.
</Callout>

Follow the [Vue 3 decorations tutorial](/guides/decorations-vue) for a complete stateful widget.

## `liveWidgetKeys`

Import: `import { liveWidgetKeys } from '@tiptap/core'`.

```ts
liveWidgetKeys(editor: Editor): ReadonlySet<string>
```

Returns the keys of all widget decorations currently live. The framework widget renderers use it internally to decide whether a widget being torn down is genuinely gone or just being reassigned to a new position (so they don't destroy a renderer whose key is still live).

<Callout title="Advanced use only" variant="hint">
  Most users won't need `liveWidgetKeys`. It exists for advanced authors building custom widget
  renderers.
</Callout>
