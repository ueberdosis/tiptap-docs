`@tiptap-pro/extension-export-docx` accepts an optional registry of **numbering format definitions** — multilevel marker style, marker text, alignment, indent, and font — selected per-list via an attribute on the outermost `<ol>`. Lists with a matching id export with the corresponding definition; lists without one export as plain `1. 2. 3.`.

The feature is opt-in. Consumers who don't supply `numberingFormats` see no behavior change.

## What the package provides — and what's on you

| Provided by the package | Provided by you |
| --- | --- |
| `numberingFormats?: NumberingFormatDefinition[]` option on `ExportDocx`, `exportDocx()`, and `ConvertConfig` | A small Tiptap extension that writes the `numberingFormat` string attribute onto the outermost `<ol>` of each multilevel list |
| `NumberingFormatDefinition` and `NumberingLevelDefinition` types | CSS that styles the marker for the editor preview |
| `LevelFormat`, `IRunOptions`, `PositiveUniversalMeasure` re-exported from `docx` | Optional UI for picking / customizing formats |

The editor-side glue is short and project-specific — write it once to fit your existing list extension and design system. The demo above is a complete reference implementation; copy it as a starting point.

## Quick start

```ts
import { Editor } from '@tiptap/core'
import { StarterKit } from '@tiptap/starter-kit'
import {
  ExportDocx,
  LevelFormat,
  type NumberingFormatDefinition,
} from '@tiptap-pro/extension-export-docx'

// 1. Define the formats your app supports.
const MY_FORMATS: NumberingFormatDefinition[] = [
  {
    id: 'decimal-paren',
    levels: [
      { baseStyle: LevelFormat.DECIMAL, textTemplate: '%1)' },
      { baseStyle: LevelFormat.LOWER_LETTER, textTemplate: '%2)' },
      { baseStyle: LevelFormat.LOWER_ROMAN, textTemplate: '%3)' },
    ],
  },
  {
    id: 'outline',
    levels: [
      { baseStyle: LevelFormat.DECIMAL, textTemplate: '%1.' },
      { baseStyle: LevelFormat.DECIMAL, textTemplate: '%1.%2.' },
      { baseStyle: LevelFormat.DECIMAL, textTemplate: '%1.%2.%3.' },
    ],
  },
]

// 2. Pass them to ExportDocx.
const editor = new Editor({
  extensions: [
    StarterKit,
    ExportDocx.configure({
      numberingFormats: MY_FORMATS,
      onCompleteExport: (blob) => {
        /* download blob */
      },
    }),
  ],
})
```

At this point the export side is ready. To make a list use a format, set `numberingFormat: 'decimal-paren'` (or any id from your registry) as an attribute on the outermost `<ol>` of the list. The next section shows the consumer-side extension that does that.

## Consumer-side extension — write your own

This ~25-line Tiptap extension adds the `numberingFormat` attribute to your existing `orderedList` node (whether it comes from `StarterKit`, `@tiptap/extension-list`, or your own list extension). It enforces "outermost only" so the attribute on a nested `<ol>` (e.g. from pasted HTML) is stripped to match the DOCX convention.

```ts
import { Extension } from '@tiptap/core'

function findOutermostOrderedList($from) {
  for (let depth = 1; depth <= $from.depth; depth += 1) {
    if ($from.node(depth).type.name === 'orderedList') {
      return { node: $from.node(depth), pos: $from.before(depth) }
    }
  }
  return null
}

export const OrderedListNumbering = Extension.create({
  name: 'orderedListNumbering',
  addGlobalAttributes() {
    return [
      {
        types: ['orderedList'],
        attributes: {
          numberingFormat: {
            default: null,
            parseHTML: (el) => {
              if (el.parentElement?.closest('ol')) return null
              return el.getAttribute('data-numbering-format')
            },
            renderHTML: (attrs) =>
              attrs.numberingFormat
                ? { 'data-numbering-format': String(attrs.numberingFormat) }
                : {},
          },
        },
      },
    ]
  },
  addCommands() {
    return {
      setOrderedListNumberingFormat:
        (id) =>
        ({ state, tr, dispatch }) => {
          const outer = findOutermostOrderedList(state.selection.$from)
          if (!outer) return false
          if (dispatch) {
            tr.setNodeMarkup(outer.pos, undefined, {
              ...outer.node.attrs,
              numberingFormat: id,
            })
          }
          return true
        },
    }
  },
})
```

Register it alongside `ExportDocx`:

```ts
const editor = new Editor({
  extensions: [
    StarterKit,
    OrderedListNumbering,
    ExportDocx.configure({ numberingFormats: MY_FORMATS, onCompleteExport: ... }),
  ],
})

editor.chain().focus().toggleOrderedList().setOrderedListNumberingFormat('outline').run()
```

## Consumer-side CSS — write your own

The exporter only cares about the attribute being on the right `<ol>`. The editor preview is yours to style. For each format, target `ol[data-numbering-format="<id>"]` and the nested levels via descendant `ol` combinators. A 2-column grid layout matches Word's hanging-indent model best:

```css
ol[data-numbering-format='decimal-paren'] {
  list-style: none;
  counter-reset: my-decimal-paren-0;
  padding-left: 0;
  margin-left: 24px; /* numberIndent: textIndent − hanging; 1 twip = 1/15 CSS px */
}
ol[data-numbering-format='decimal-paren'] > li {
  display: grid;
  grid-template-columns: minmax(24px, max-content) 1fr; /* col1 = hanging */
  column-gap: 0;
  align-items: baseline;
  counter-increment: my-decimal-paren-0;
}
ol[data-numbering-format='decimal-paren'] > li > * {
  grid-column: 2; /* body content + nested lists in col2 */
}
ol[data-numbering-format='decimal-paren'] > li::marker {
  content: none;
}
ol[data-numbering-format='decimal-paren'] > li::before {
  content: counter(my-decimal-paren-0, decimal) ')';
  text-align: left;
  white-space: nowrap;
}
```

Repeat for nested depths via `ol[...] ol` selectors, advancing the counter name (`my-decimal-paren-1`, `my-decimal-paren-2`) and the marker text (e.g. `lower-alpha`, `lower-roman`).

For dynamic registries (e.g. a customize-format dialog where the user adds their own definitions at runtime), it's worth writing a small generator that walks `NumberingFormatDefinition[]` and produces this CSS. The demo above ships such a generator (`numbering-format-css.js`) — vanilla JS with no dependencies. Copy it as a starting point and call `generateNumberingFormatCSS(formats)` at boot.

## `NumberingFormatDefinition`

```ts
interface NumberingFormatDefinition {
  id: string
  levels: NumberingLevelDefinition[]
}
```

| Field | Type | Description |
| --- | --- | --- |
| `id` | `string` | Unique within your `numberingFormats[]`. Serialized into the orderedList's `numberingFormat` attribute. |
| `levels` | `NumberingLevelDefinition[]` | One entry per nesting depth. Must be non-empty. When a list nests deeper than the array, depth `N` reuses `levels[N % levels.length]`. |

## `NumberingLevelDefinition`

```ts
interface NumberingLevelDefinition {
  baseStyle: LevelFormat
  textTemplate: string
  startAt?: number
  alignment?: 'left' | 'center' | 'right'
  numberIndent?: number | PositiveUniversalMeasure
  textIndent?: number | PositiveUniversalMeasure
  markerFont?: Omit<IRunOptions, 'text' | 'children'>
}
```

| Field | Default | Description |
| --- | --- | --- |
| `baseStyle` | (required) | Any docx [`LevelFormat`](https://docx.js.org/api/enums/LevelFormat.html) value (`DECIMAL`, `DECIMAL_ZERO`, `UPPER_LETTER`, `LOWER_LETTER`, `UPPER_ROMAN`, `LOWER_ROMAN`, `NONE`, etc.). |
| `textTemplate` | (required) | Marker text using Word's `<w:lvlText>` grammar — see below. |
| `startAt` | `1` | Initial counter value. |
| `alignment` | `'left'` | Marker text alignment within the number area. |
| `numberIndent` | Word's per-level default | Distance from the page margin to the marker. Twips number or `PositiveUniversalMeasure` string (`'0.63cm'`, `'0.25in'`, `'18pt'`). |
| `textIndent` | Word's per-level default | Distance from the page margin to the body text. Should be greater than `numberIndent`. |
| `markerFont` | — | Marker-only run formatting. Survives DOCX export; editor preview is yours to wire. |

## The `textTemplate` grammar

- `%1` through `%9` reference the counter at the 1-indexed nesting depth. So `%1` is always "the counter at the outermost level", regardless of which level the `textTemplate` belongs to.
- All other characters render literally.
- Stray `%` followed by a non-digit (e.g. `"50%"`) is preserved.

To make each level display its own counter, use ascending `%N` per level. To make a level include parent counters (legal-style outlines), chain them: `'%1.%2.'` at level 1 renders `'1.1.'`.

| Template at level N | Rendered example |
| --- | --- |
| `'%1.'` at level 0 | `1.` |
| `'%2.'` at level 1 | `1.` (the level-1 counter) |
| `'%1.%2.'` at level 1 | `1.1.` |
| `'%1.%2.%3.'` at level 2 | `1.1.1.` |
| `'Article %1'` at level 0 | `Article 1` |
| `'§ %1 —'` at level 0 | `§ 1 —` |
| `'(%3)'` at level 2 | `(1)` |

## Schema convention — outermost only

The `numberingFormat` attribute belongs on the **outermost** `<ol>` only. One definition declares all nesting levels for the entire multilevel list. Your consumer-side extension should enforce this on parse and on command application — see the example above.

All nesting levels under one outermost `<ol>` share a single counter scope, and sub-levels restart when the parent advances, exactly as Word does.

## Cycling rule

`levels.length` defines the cycle period. When a list nests deeper than `levels.length - 1`, depth `N` reuses `levels[N % levels.length]` for both DOCX export and your editor preview. Supply nine levels to cover Word's full nesting depth without cycling; supply fewer when the deeper levels can naturally repeat the shallower entries.

## Defaults match Word's standard

When `numberIndent` / `textIndent` are omitted, the exporter emits Word's standard multilevel-list defaults:

| Depth | `textIndent` | `numberIndent` | Hanging |
| --- | --- | --- | --- |
| 0 | `720` (0.50″) | `360` (0.25″) | `360` |
| 1 | `1140` (0.79″) | `780` (0.54″) | `360` |
| 2 | `1440` (1.00″) | `1080` (0.75″) | `360` |
| 3 | `1740` (1.21″) | `1380` (0.96″) | `360` |
| 4 | `2040` (1.42″) | `1680` (1.17″) | `360` |
| 5 | `2340` (1.63″) | `1980` (1.38″) | `360` |
| 6 | `2640` (1.83″) | `2280` (1.58″) | `360` |
| 7 | `2940` (2.04″) | `2580` (1.79″) | `360` |
| 8 | `3240` (2.25″) | `2880` (2.00″) | `360` |

Use these same values in your editor preview CSS so the layout matches the exported `.docx` exactly.

## Resolution rules

- A matched id renders every level of that multilevel list — including all nested `<ol>`s — using the corresponding definition.
- An unmatched id (typo, missing attribute, empty `numberingFormats`) renders as plain `1. 2. 3.` numbering.
- Sibling multilevel lists referencing the same format restart independently — each starts at its own `startAt`.

## Known limitations

Word features the exporter intentionally does not model:

| Feature | Workaround / scope |
| --- | --- |
| Forcing parent-counter refs to render as arabic regardless of base style (Word's "legal numbering" override) | Define a separate format with all-`DECIMAL` levels for the same visual output. |
| Per-level counter-restart customization | Word's default behavior (sub-levels restart when the parent advances) is always used. |
| Marker-text suffix (tab / space / nothing between marker and body) | Always `tab` (Word's default). |
| Continuing numbering across separate lists | Each list starts at its `startAt` independently. |
| Per-item marker overrides | Use a separate list to start at a different counter value. |
| DOCX → editor import | Handled separately by `@tiptap-pro/extension-import-docx`. |
