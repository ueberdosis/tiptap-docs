In Word, headings are paragraphs with a heading style applied (Heading 1 through Heading 9). Tiptap represents them as `heading` nodes with a `level` attribute.

## What you need

- **Extensions:** `StarterKit` (includes `Heading`) and `TextAlign` for alignment support
- **Configuration:** `TextAlign.configure({ types: ['heading', 'paragraph'] })`

<Callout title="Heading styling requires configuration" variant="hint">
  Heading levels and alignment round-trip out of the box. Visual styling (font, color, spacing) is extracted during import and included in the JSON, but the default Heading extension does not render these attributes. The editor renders headings with your CSS, not Word's theme styles. See the [Styling converted content](/conversion/getting-started/guides/styling-converted-content) guide for how to extend the Heading extension and control heading appearance.
</Callout>

## Support overview

| | Import | Editor | Export |
| --- | --- | --- | --- |
| Heading levels 1 to 6 | Supported | Supported | Supported |
| Title and Subtitle styles | Mapped to level 1 and 2 | Supported | Exported as level 1 and 2 |
| Heading levels 7 to 9 | Supported | Not supported | Not supported | Import produces heading nodes with levels 7–9, but the editor Heading extension only supports 1–6 by default. A custom extension is needed to render these levels. |
| Custom heading styles | Resolved to closest standard level | Style name lost | Generic heading style applied |
| Text alignment | Supported | Requires TextAlign extension | Supported |
| Inline formatting | Supported | Supported | Supported |
| Heading numbering | Converted to indent attr | Not rendered | Not preserved |
| Visual styling (font, color, spacing) | Converted to attrs | Not in schema | Export defaults applied |

## Import

Import headings using the [editor extension](/conversion/import/docx/editor-extension) or the [REST API](/conversion/import/docx/rest-api). Both produce identical heading output.

The conversion service detects Word heading styles (`Heading1` through `Heading9`, plus `Title` and `Subtitle`) and produces a `heading` node. Custom heading styles (e.g., "Company Heading" based on Heading 2) are resolved by walking the `basedOn` style chain. The resolved level is preserved, but the custom style name is discarded.

**Title** maps to level 1. **Subtitle** maps to level 2. These styles are common in corporate Word templates, so heading levels in the editor may differ from what users expect if their documents use Title/Subtitle instead of Heading 1/Heading 2.

<CodeDemo isPro path="/Extensions/ContentTypeHeadingsImport?hideSource=true" />

The conversion service outputs heading nodes in this structure:

```json
{
  "type": "heading",
  "attrs": {
    "level": 2,
    "textAlign": "center"
  },
  "content": [
    {
      "type": "text",
      "text": "My Heading",
      "marks": [
        { "type": "bold", "attrs": {} }
      ]
    }
  ]
}
```

<Callout variant="warning" title="Heading levels 7–9">
  Word supports heading levels 1–9, and the import converter recognizes all nine levels. However, Tiptap's default Heading extension only supports levels 1–6. Heading nodes with levels 7–9 will be produced on import but require a custom extension to render in the editor. Without one, the schema will drop them to paragraphs when the document loads.
</Callout>

### What's preserved

- Heading level (1–6)
- Inline formatting within the heading (bold, italic, color, and font as marks on child text nodes)
- Text alignment (as `textAlign` attribute)
- Paragraph spacing before/after (as `marginTop`/`marginBottom` attributes)

### What's lost

The heading node schema only defines `level` as an attribute. All other imported attributes (font size, font family, colors, margins, borders, spacing, indentation) are not part of the heading node schema and require a [custom extension](/editor/extensions/custom-extensions/extend-existing) to preserve. Word heading numbering (common in legal and corporate documents) produces an `indent` attribute that is also not rendered or exported.

To prevent silent loss of these attributes, see the [invalid schema guide](/guides/invalid-schema). To render them, you can [extend the heading extension](/editor/extensions/custom-extensions/extend-existing) with custom attributes.

## Editor rendering

The [`Heading`](/editor/extensions/nodes/heading) extension (included in `StarterKit`) renders heading nodes as `<h1>` through `<h6>` HTML tags based on the `level` attribute. No separate installation is needed for basic heading support.

<CodeDemo isPro path="/Extensions/ContentTypeHeadingsEditor?hideSource=true" />

### Text alignment requires an additional extension

The Heading extension does **not** include text alignment. To preserve alignment from imported DOCX headings, install `TextAlign` and configure it to include headings:

```ts
import TextAlign from '@tiptap/extension-text-align'

TextAlign.configure({
  types: ['heading', 'paragraph'],
})
```

Without TextAlign, the `textAlign` attribute from import is not recognized by the schema. See the [invalid schema guide](/guides/invalid-schema) for how to manage unrecognized attributes.

### Visual differences from Word

The editor renders headings with your CSS styles, not Word's theme styles. A Heading 2 in Word (typically Aptos Light, 13pt, blue) will look different in the editor unless you style `h2` to match. Structure is preserved. Visual appearance is yours to control through your editor's CSS. See the [Styling converted content](/conversion/getting-started/guides/styling-converted-content) guide for how to manage this.

## Export

Export headings using the [editor extension](/conversion/export/docx/editor-extension) or the [REST API](/conversion/export/docx/rest-api). The extension supports [`paragraphOverrides` and `textRunOverrides`](/conversion/export/docx/styles#element-overrides) for fine-grained control over heading paragraph and text run defaults. The REST API does not accept element overrides. To customize heading styles via the REST API, use [`styleOverrides`](/conversion/export/docx/styles) to redefine the heading document styles.

<CodeDemo isPro path="/Extensions/ContentTypeHeadingsExport?hideSource=true" />

The export converter takes a `heading` node and produces a DOCX paragraph with the corresponding heading style. Text alignment (`left`, `center`, `right`, `justify`) is exported correctly.

**Level mapping:** `level: 1` → Word Heading 1, `level: 2` → Word Heading 2, through level 6.

### Customizing exported heading styles

The export applies opinionated default styles to each heading level: Aptos Light font, blue color (#2E74B5), and level-specific sizes. If you are exporting documents for a corporate template or brand, these defaults will likely not match your requirements. Customize them via `styleOverrides`:

```ts
ExportDocx.configure({
  styleOverrides: {
    paragraphStyles: [
      {
        id: 'Heading1',
        name: 'Heading 1',
        run: {
          font: 'Arial',
          size: 32,        // half-points (32 = 16pt)
          bold: true,
          color: '000000',
        },
        paragraph: {
          spacing: {
            before: 240,   // twips (240 = 12pt)
            after: 120,    // twips (120 = 6pt)
          },
        },
      },
    ],
  },
})
```

### What round-trips

A Heading 2, center-aligned, with bold and colored text inside will survive a full import → edit → export cycle. The heading level, alignment, and inline formatting are all preserved.

What changes: the exported heading will use the export's default (or overridden) styles rather than the original DOCX's theme styles. Font, size, spacing, and color are determined by the export configuration, not carried over from the original document.
