Tables convert between DOCX and Tiptap using the `table`, `tableRow`, `tableCell`, and `tableHeader` node types. Basic structure, colspan, and column widths round-trip cleanly. Header rows import correctly but are not preserved on export. Cell-level styling is lossy on export.

## What you need

- **Without Pages:** [`@tiptap/extension-table`](/editor/extensions/nodes/table) (open source TableKit, not in StarterKit)
- **With Pages:** [`@tiptap-pro/extension-pages-tablekit`](/pages/guides/table-with-pages) instead. This version uses CSS Grid layout so tables can split across page breaks. Do not install both.
- **Configuration:** None required for basic table support.

Both extensions register the same node types (`table`, `tableRow`, `tableCell`, `tableHeader`) with the same schema, so imported DOCX content works with either.

## Support overview

| Feature | Import | Editor | Export | Notes |
| --- | --- | --- | --- | --- |
| Basic tables | Supported | Supported | Supported | Fixed layout in DOCX output |
| Header rows | Supported | Supported | Not preserved | Uses `tableHeader` node type in editor, but export creates all rows identically |
| Colspan | Supported | Supported | Supported | Full round-trip |
| Rowspan | Supported | Supported | Supported | A pre-scan across all rows builds a complete rowspan map before processing cells |
| Column widths | Supported | Supported | Supported | Stored as `colwidth` array in pixels, converted to twips on export |
| Cell backgrounds | Supported | Supported | Not preserved | `background` captured on import, not read on export |
| Cell borders | Supported | Supported | Not preserved | Border attrs captured on import, not read on export |
| Global table styling | N/A | N/A | Supported | Via `tableOverrides` and `tableCellOverrides` |

## Import

Import tables using the [editor extension](/conversion/import/docx/editor-extension) or the [REST API](/conversion/import/docx/rest-api). Both produce identical output.

<CodeDemo isPro path="/Extensions/ContentTypeTablesImport?hideSource=true" />

The conversion service reads DOCX tables and produces `table`, `tableRow`, and `tableCell` or `tableHeader` nodes. Column widths are stored in the `colwidth` attribute as an array of pixel values. Cell backgrounds and borders are captured as node attributes.

Merged cells using colspan convert reliably. Rowspan is also fully supported. The converter performs a pre-scan across all rows to build a complete rowspan map before processing cells, correctly resolving Word's `vMerge` "continue" markers.

If your editor schema does not include table nodes, imported table content will be dropped. See the [invalid schema guide](/guides/invalid-schema) for handling strategies.

## Editor rendering

The [`TableKit`](/editor/extensions/nodes/table) extension provides table, row, cell, and header cell nodes. It is not included in StarterKit and must be installed separately. If you are using the [Pages extension](/pages/getting-started/overview), use the [Pages-compatible TableKit](/pages/guides/table-with-pages) instead.

<CodeDemo isPro path="/Extensions/ContentTypeTablesEditor?hideSource=true" />

Install the table extension that matches your setup:

```bash
# Standard editor (without Pages)
npm install @tiptap/extension-table

# With the Pages extension
npm install @tiptap-pro/extension-pages-tablekit
```

```ts
// Use one or the other, not both
import { TableKit } from '@tiptap/extension-table'
// OR
import { TableKit } from '@tiptap-pro/extension-pages-tablekit'

const editor = new Editor({
  extensions: [
    StarterKit,
    TableKit,
  ],
})
```

Tables render as HTML `<table>`, `<tr>`, `<td>`, and `<th>` elements. Column resizing is supported when `resizable: true` is set on the Table extension. The `colwidth` attribute from import controls initial column widths.

## Export

Export tables using the [editor extension](/conversion/export/docx/editor-extension) or the [REST API](/conversion/export/docx/rest-api). The extension supports [`tableOverrides` and `tableCellOverrides`](/conversion/export/docx/styles#element-overrides) for global table styling defaults. The REST API does not accept these overrides.

<CodeDemo isPro path="/Extensions/ContentTypeTablesExport?hideSource=true" />

The exporter produces DOCX tables with fixed layout. It reads the `colwidth` attribute and converts pixel values to twips. If no `colwidth` is set, columns are distributed evenly across the available page width.

Header rows are not preserved. The exporter creates all rows identically with `new TableRow({ children: cells })` and does not set `tableHeader: true` on the DOCX row based on the `tableHeader` node type. Header-row semantics are lost in the output. Merged cells (colspan and rowspan) export correctly.

<Callout title="Cell-level styling is not exported" variant="warning">
  Cell backgrounds (`background`) and per-cell borders are not read during export. All cells get default single-line borders. If you need custom table styling in the output, use [`tableOverrides` and `tableCellOverrides`](/conversion/export/docx/styles#element-overrides) on the editor extension for global defaults. These overrides are not available via the REST API.
</Callout>

### Override system

The export extension accepts two options for global table styling:

```ts
ExportDocx.configure({
  tableOverrides: {
    borders: {
      top: { style: 'single', size: 4, color: '000000' },
      bottom: { style: 'single', size: 4, color: '000000' },
      left: { style: 'single', size: 4, color: '000000' },
      right: { style: 'single', size: 4, color: '000000' },
      insideHorizontal: { style: 'single', size: 4, color: '000000' },
      insideVertical: { style: 'single', size: 4, color: '000000' },
    },
  },
  tableCellOverrides: {
    shading: { fill: 'F5F5F5' },
  },
})
```

These overrides apply globally to all tables in the exported document. Per-cell styling is not supported on export.
